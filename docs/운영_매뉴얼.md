## 2. Git 설치 및 초기 설정

### 2-1. Git 설치 (이미 완료)

1. https://git-scm.com 접속
2. **Download for Windows** 클릭
3. 설치 마법사에서 **모두 Next** → **Install** 클릭
4. 설치 완료 후 터미널 재시작

### 2-2. Git 사용자 정보 설정 (최초 1회)

터미널을 열고 아래 두 줄을 입력합니다.
`홍길동`과 이메일은 본인 정보로 바꾸세요.

```bash
git config --global user.name "홍길동"
git config --global user.email "hong@korea.go.kr"
```

확인:
```bash
git config --list
```
`user.name`과 `user.email`이 보이면 완료입니다.

### 2-3. Git 핵심 개념 (3가지만 기억)

```
[내 PC 소스코드]  →  git add  →  git commit  →  git push  →  [GitHub]
     수정완료         저장 준비      저장 확정       업로드
```

| 명령어 | 비유 | 설명 |
|---|---|---|
| `git add 파일명` | 택배 박스에 담기 | 변경된 파일을 업로드 대상으로 선택 |
| `git commit -m "설명"` | 택배 박스 봉인 | 변경 내용을 확정하고 메모 작성 |
| `git push` | 택배 발송 | GitHub에 업로드 |
| `git pull` | 택배 수령 | GitHub에서 최신 내용 받기 |
| `git status` | 택배 현황 조회 | 현재 변경된 파일 목록 확인 |

### 5-2. Docker 실행 확인

작업 표시줄에 고래 아이콘(🐳)이 보이면 실행 중입니다.

```bash
docker --version
docker ps
```

오류 없이 실행되면 완료입니다.

### 5-3. Docker 핵심 개념

```
[소스코드 + requirements.txt]
         ↓  docker build
     [이미지]  ← 프로그램이 담긴 압축파일 같은 것
         ↓  docker run
     [컨테이너]  ← 실제로 실행 중인 프로그램
```

| 명령어 | 설명 |
|---|---|
| `docker build -t 이름 .` | 이미지 만들기 |
| `docker run 이름` | 컨테이너 실행 |
| `docker ps` | 실행 중인 컨테이너 목록 |
| `docker stop 컨테이너ID` | 컨테이너 중지 |
| `docker-compose up -d` | docker-compose.yml로 전체 실행 |
| `docker-compose down` | 전체 중지 |

## 7. CI/CD 파이프라인 이해

### 7-1. CI/CD란?

```
개발자가 코드 수정 → git push
         ↓
    [GitHub Actions 자동 실행]
         ↓
    CI (코드 검사)          CD (배포)
    ├─ 린트 검사            ├─ Docker 이미지 빌드
    ├─ 보안 검사            ├─ Docker Hub에 업로드
    └─ Docker 빌드 테스트   └─ 서버에 자동 배포
```

**CI** = Continuous Integration (지속적 통합) — 코드 품질 자동 검사
**CD** = Continuous Deployment (지속적 배포) — 서버 자동 배포

### 7-2. 현재 파이프라인 파일 위치

```
.github/
  workflows/
    ci.yml   ← develop 브랜치에 push할 때 자동 실행
    cd.yml   ← main 브랜치에 merge할 때 자동 실행 (배포)
```

### 7-3. CI 실행 결과 확인 방법

**방법 1 — GitHub 웹사이트**
1. `https://github.com/forevergoodnessNa/acrc-rag-chatbot` 접속
2. **Actions** 탭 클릭
3. 최근 실행 결과 확인
   - 🟢 초록 체크 = 성공
   - 🔴 빨간 X = 실패 (클릭하면 오류 내용 확인 가능)

**방법 2 — 터미널**
```bash
gh run list --limit 5
```

### 7-4. CI 실패 시 대처

```bash
# 실패한 run ID 확인 후 상세 로그 보기
gh run view [RUN_ID] --log-failed
```

## 8. 일상 운영 — 소스 수정 후 반영하기

### 8-1. 기본 작업 흐름

```
1. 소스 수정
2. 변경 내용 확인
3. 스테이징 (담기)
4. 커밋 (확정)
5. 푸시 (업로드)
6. CI 결과 확인
```

### 8-2. 단계별 명령어

```bash
# 프로젝트 폴더로 이동
cd C:\NuriNamuAiChat

# 1단계: 어떤 파일이 변경됐는지 확인
git status

# 2단계: 변경 내용 상세 확인 (선택사항)
git diff

# 3단계: 변경된 파일 스테이징 (특정 파일)
git add server/api_server.py

# 또는 전체 파일 한번에
git add .

# 4단계: 커밋 (변경 이유를 간단히 적기)
git commit -m "수정 내용을 한 줄로 설명"

# 예시:
git commit -m "fix: 로그인 오류 수정"
git commit -m "feat: 새 기능 추가"
git commit -m "docs: 매뉴얼 업데이트"

# 5단계: GitHub에 업로드
git push origin develop

# 6단계: CI 결과 확인 (30초~1분 후)
gh run list --limit 1
```

### 8-3. 커밋 메시지 작성 규칙

| 접두어 | 의미 | 예시 |
|---|---|---|
| `feat:` | 새 기능 추가 | `feat: 파일 업로드 기능 추가` |
| `fix:`  | 버그 수정 | `fix: 답변 오류 수정` |
| `docs:` | 문서 수정 | `docs: 매뉴얼 업데이트` |
| `style:` | 화면 디자인 수정 | `style: 버튼 색상 변경` |
| `refactor:` | 코드 구조 개선 | `refactor: 검색 로직 개선` |

### 8-4. 실수로 잘못 커밋했을 때

```bash
# 마지막 커밋 취소 (파일 변경은 유지)
git reset --soft HEAD~1

# 수정 후 다시 커밋
git add .
git commit -m "올바른 메시지"
git push origin develop
```

## 10. 문제 발생 시 대처

### 10-1. "모델을 찾을 수 없습니다" 오류

```
오류: model 'qwen2.5:3b' not found
```

**해결:**
```bash
ollama pull qwen2.5:3b
ollama pull exaone3.5:2.4b
```

### 10-2. "메모리 부족" 오류

```
오류: model requires more system memory than is available
```

**해결:** `.env` 파일에서 더 작은 모델로 변경
```
MAIN_LLM_MODEL=qwen2.5:3b       ← 현재 설정 (1.9GB)
ROUTER_LLM_MODEL=exaone3.5:2.4b ← 현재 설정 (1.5GB)
```

### 10-3. "연결 오류" (Read timed out)

**원인:** 모델이 메모리에 아직 로드되지 않음 (콜드 스타트)
**해결:** 서버 시작 후 1~2분 대기 후 다시 시도

### 10-4. DB 연결 오류

```bash
# Docker DB 컨테이너 상태 확인
docker ps

# DB 컨테이너가 없으면 시작
docker-compose up -d pgvector
```

### 10-5. CI 빌드 실패

```bash
# 실패 원인 확인
gh run list --limit 3
gh run view [RUN_ID] --log-failed

# 로컬에서 린트 검사 먼저 실행
venv\Scripts\ruff check . --select=E,F,W --ignore=E501
```

### 10-6. git push 거부 (rejected)

```
오류: Updates were rejected because the remote contains work that you do not have locally
```

**해결:**
```bash
git pull origin develop
# 충돌이 없으면 자동 병합됨
git push origin develop
```
